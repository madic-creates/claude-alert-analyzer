- **fix: RunAgenticDiagnostics called with IP instead of hostname** — `processAlert` in `cmd/checkmk-analyzer/main.go` was passing `hostAddress` (the IP from the webhook) to `RunAgenticDiagnostics` instead of `hostname`. The function dials SSH using its argument as the target address; when `known_hosts` uses hostnames (the typical deployment), SSH host key verification fails because the connected IP doesn't match any entry. The fix passes `hostname`, which was already validated against CheckMK's known IP via `ValidateAndDescribeHost`. One-line change: `hostAddress` → `hostname`.
- **fix: hostname-vs-IP bug re-introduced in pipeline.go refactor + regression test** — The recent design-review refactoring moved alert processing from `cmd/checkmk-analyzer/main.go` into a new `internal/checkmk/pipeline.go`, but the call to `RunAgenticDiagnostics` used `hostAddress` (IP) instead of `hostname` again. Fixed the one-line bug. To prevent recurrence, extracted `Dialer` interface from concrete `*SSHDialer` (in `ssh.go`); updated `RunAgenticDiagnostics`, `PipelineDeps`, and `main.go` accordingly. Added `TestProcessAlert_SSH_UsesHostnameNotIP` in `pipeline_test.go` which wires a `mockDialer` that records its argument, asserting it is the hostname not the IP.
- **fix: bound Prometheus response body size in promqlQuery** — `promqlQuery` in `internal/k8s/context.go` was the only API response reader not using `io.LimitReader`. Every other HTTP read in the codebase (CheckMK client, Claude client) already capped reads at `shared.MaxResponseBytes` (2 MiB), but the Prometheus path used bare `io.ReadAll(resp.Body)`. A misbehaving or misconfigured Prometheus server returning a huge payload (e.g. a high-cardinality query result) could cause unbounded memory allocation. Fixed with `io.LimitReader(resp.Body, shared.MaxResponseBytes)`. Added `TestGetPrometheusMetrics_OversizedResponse` regression test.
- **fix: RunToolLoop nil-content message on max_tokens stop_reason** — `RunToolLoop` in `internal/shared/claude.go` only handled `stop_reason == "end_turn"` explicitly; any other stop reason (notably `"max_tokens"`) fell through to the tool-processing path. With no `tool_use` blocks, `toolResults` stayed nil and was appended as `ToolMessage{Content: nil}`, which marshals to `"content": null`. The Anthropic API rejects this with a 400 error, causing the entire agentic diagnosis to fail when the model simply hits the token limit. Fixed by returning `extractText(resp.Content)` immediately when `len(toolResults) == 0` after the loop. Added `TestRunToolLoop_MaxTokensStopReason` regression test verifying exactly one API call and the partial text result.
- **security: validate namespace before PromQL interpolation** — `GetMetrics` in `internal/k8s/context.go` interpolated `alert.Labels["namespace"]` directly into PromQL query strings (three queries: CPU, memory, pod restarts). A webhook payload with a malicious namespace value like `default"} or up{namespace="` could break out of the label-value string and modify the query sent to Prometheus. Added `isValidNamespace()` (validates against Kubernetes namespace name rules: lowercase alphanumeric + hyphens, ≤63 chars) and call it before using the namespace in queries; invalid namespaces are logged and the three namespace-scoped queries are skipped. Added `TestIsValidNamespace` (14 valid/invalid cases) and `TestGetMetrics_MaliciousNamespaceDroppedFromQuery` (confirms the injected token never appears in recorded Prometheus requests) in `context_test.go`.
- **security: normalize argv[0] before denylist check in isDenied** — `isDenied` in `internal/checkmk/agent.go` checked `argv[0]` literally, so an absolute path like `["/bin/rm", "-rf", "/"]` bypassed the denylist (checked against `"rm"`) entirely. An LLM producing absolute-path commands would have been able to run any denied command. Fixed with `filepath.Base(argv[0])` to normalize `/bin/rm` → `rm`, `./rm` → `rm`, `../bin/sudo` → `sudo`, etc., before the map lookup. The `systemctl` special-case path is equally fixed. Added 3 regression tests: `TestIsDenied_AbsolutePathBypassBlocked`, `TestIsDenied_RelativePathBypassBlocked`, `TestIsDenied_AbsolutePathSystemctl`.
- **fix: panic recovery in worker pool** — Worker goroutines in `Server.Run()` called `s.process()` directly with no panic protection. A panic inside any alert processing path would permanently kill that goroutine, silently shrinking the worker pool with no recovery. Extracted `safeProcess()` (unexported method on `Server`) that wraps `s.process` with a deferred `recover()` + `slog.Error` log. Workers now call `safeProcess()` instead. Added `TestServer_SafeProcess_PanicDoesNotPropagate` which verifies that calling `safeProcess` twice — first with a panicking process, then with a normal one — recovers cleanly and the second call executes successfully.
- **fix: expose processing duration as Prometheus summary type** — `MetricsHandler()` in `metrics.go` exposed `alert_analyzer_processing_duration_seconds_sum` and `alert_analyzer_processing_duration_seconds_count` as two independent `counter`-typed metrics, each with their own `# HELP` and `# TYPE` lines. The `_sum` and `_count` suffixes are reserved by the Prometheus data model for histogram/summary sub-samples; giving them standalone `counter` types causes `promtool check metrics` lint errors and confuses Prometheus tooling (e.g. Grafana's explore view, recording rules that aggregate by type). Fixed by emitting a single `# HELP`/`# TYPE` block for the base name `alert_analyzer_processing_duration_seconds` with type `summary`, followed by the `_sum` and `_count` sample lines (no per-sub-sample headers). Added `TestMetricsHandler_ProcessingDurationIsSummary` which asserts: `summary` type line present, no separate `TYPE` lines for `_sum`/`_count`, and correct numeric values.

These are the first tests for `RunAgenticDiagnostics` itself — the core agentic diagnostic function was previously only tested indirectly via `pipeline_test.go`.

- **fix: isDenied skips flags before systemctl subcommand** — `isDenied` in `internal/checkmk/agent.go` checked `argv[1]` directly as the systemctl subcommand. When Claude (reasonably) generates `["systemctl", "--no-pager", "status", "nginx"]`, `argv[1]` is `"--no-pager"`, which is not in `systemctlReadOnly`, so the command was wrongly denied — even though the agent system prompt explicitly lists `systemctl status/show` as allowed. Fixed by iterating `argv[1:]` and skipping leading dash-prefixed arguments to find the first non-flag token as the subcommand; bare `systemctl` with only flags (no subcommand) is still denied. Added `TestIsDenied_SystemctlFlagsBeforeSubcommand` with 9 subtests covering `--no-pager` + allowed subcommand, `--user` + allowed subcommand, flags + denied subcommand, and flags-only (no subcommand).
- **fix: sort Prometheus metric labels for deterministic output** — `query` in `internal/k8s/context.go` built each result line by iterating `r.Metric` (a `map[string]string`) directly. Go map iteration is random, so identical alerts produced different label orderings on successive runs, making the context injected into Claude's prompt non-deterministic and reducing consistency of AI analysis. Fixed with `sort.Strings(labels)` before joining. Added `TestQuery_MetricLabelsAreSorted` which returns a three-label result with keys intentionally out of alphabetical order and asserts `alpha` precedes `middle` precedes `zoo` in the output.
- **test: GetHostServices direct coverage** — `GetHostServices` in `internal/checkmk/context.go` was only exercised indirectly through `GatherContext` tests, all of which used a mock returning an empty service list. - **security: redact secrets from CheckMK service plugin_output** — `GetHostServices` in `internal/checkmk/context.go` formatted service check output (`plugin_output`) directly into the Claude prompt without running it through `shared.RedactSecrets`. This was inconsistent with k8s pod logs and SSH command output, both of which are redacted before being forwarded to the Claude API. A poorly-written CheckMK check plugin can emit credentials (DB connection strings, API keys, passwords) in its check output; without redaction those would be sent to Claude unmasked. Fixed by applying `shared.RedactSecrets(svc.Extensions.Output)` before formatting each service line. Added `TestGetHostServices_SecretsRedactedInOutput` regression test (password and API key patterns in plugin_output are redacted; service name and state are preserved).
- **security: redact secrets from GatherContext service_output and perf_data** — `GatherContext` in `internal/checkmk/context.go` embedded `alert.Fields["service_output"]` and `alert.Fields["perf_data"]` (both from the incoming webhook payload) into the Claude prompt without calling `shared.RedactSecrets`. This was inconsistent with `GetHostServices`, which already redacts `plugin_output` from the same host — both fields are check plugin outputs that can contain credentials. Fixed by wrapping both field values with `shared.RedactSecrets()` in the `fmt.Sprintf` call. Added `TestGatherContext_ServiceOutputSecretsRedacted` regression test verifying that passwords and tokens in `service_output`/`perf_data` are redacted while service metadata is preserved.
